# Performance (Core Web Vitals)

Score: 48/100

**Method:** Lighthouse 13.4.1 CLI (lab data) run against the live production site, 3 URLs × 2 form factors = 6 audits: homepage (`/`), category page (`/category/shipping-containers`), product page (`/product/1000-gallon-underground-propane-tanks`), each on mobile (DevTools throttling, 4G/4x CPU slowdown) and desktop (simulated throttling). No Google API/CrUX credentials were configured in this environment, so field data (28-day CrUX) could not be pulled — all figures below are lab data. Lighthouse itself flagged the test host's CPU as slower than its calibration baseline on one mobile run, which inflates *absolute* millisecond values on mobile somewhat; the diagnostic root causes (which resource is responsible, why) are unaffected by this and are the basis for the findings below.

**Core Web Vitals status (lab, mobile — the surface Google weights most heavily):**
- LCP: **Fail** on all 3 page types (10.6s–18.5s vs. ≤2.5s "Good" threshold) — well into "Poor."

## What Works

- CLS is excellent everywhere tested (0–0.007, well inside the ≤0.1 "Good" threshold) — images and layout containers are properly dimensioned and nothing is shifting content after paint.
- No third-party script bloat: Lighthouse's third-party audit returned zero flagged entities on any page — the main thread isn't being hijacked by ad tags, chat widgets, or analytics scripts.
- Font loading isn't causing FOIT/CLS: font-display audit scored 1/1 with 0ms wasted savings across all pages.
- TTFB is fast and consistent (~110–120ms server response time on every page/device) — Vercel's SSR/edge response isn't a bottleneck.
- **Self-hosted product images are confirmed loading correctly**: every image request across all 6 audit runs (including the product page's 15 images) returned HTTP 200. The July 19 402-billing outage (Vercel image optimizer blocking every product photo) is resolved, and the commit d4951fe self-host migration did not reintroduce broken images.
- DOM size is lean (~257 elements on the product page) and render-blocking resources are minimal (a single ~11KB stylesheet, ~150ms block).
- Cache headers on static/image assets score well (cache-lifetime audit: 1/1).

## Findings

**next.config.ts still sets `images.unoptimized: true` — self-hosted images bypass Next.js Image Optimization entirely** | Critical | Confirmed by direct inspection of `next.config.ts` (lines 19–30). The `unoptimized: true` flag was left in place from the earlier Vercel image-optimizer billing workaround, and its accompanying code comment ("hotlinked from cdn.shopify.com/conexdepotshipping.com") is now stale — commit d4951fe already restored self-hosted product images, but the optimizer flag was never reverted to match. As a result every image on the site is served exactly as uploaded — full-resolution JPEG/PNG, no responsive `srcset`, no WebP/AVIF conversion — regardless of the `next/image` component being used in markup. Lighthouse's image-delivery insight scored 0 (fail) on 5 of 6 audit runs. Concrete evidence: the product hero image on `/product/1000-gallon-underground-propane-tanks` (`conexdepotshipping-1000-gallon-underground-propane-tank-for-sale-354aa85c.jpg`) is 742KB, served at its native 2320×1740px into a 662×496px display box — 681KB (92%) of that transfer is pure waste from missing responsive sizing, plus ~69KB more recoverable by switching to WebP/AVIF. The site header logo (`logo-mark-256.png`, 50KB) is served at 256×256px into a 63×63px slot — 49.7KB (99%) wasted, and it loads on every single page of the site. The homepage hero background (`cdn-untitled-1-1-549dd1ae.png`) is a 473KB unconverted PNG. Total page weight is inflated to 3.2–3.5MB on the product/homepage and ~0.9–1.2MB on the category page, almost entirely from images. | Remove `images.unoptimized: true` now that images are self-hosted — the original 402-billing trigger (re-encoding hotlinked third-party CDN images on every request) no longer applies once images are optimized once at Vercel's edge and cached, not re-billed per hotlink. Pair with pre-resizing source images to sane max dimensions (~1600px) and using `next/image`'s `sizes` prop so it emits correct responsive `srcset`s. Expected impact: 60–90% image payload reduction site-wide — the single highest-leverage fix available for both LCP and page-weight-driven mobile bounce/conversion loss.

**LCP fails "Good" threshold on all three page types on mobile, driven by the same unoptimized-image root cause** | Critical | Lab LCP (mobile): homepage 17.5s, category 10.6s, product 18.5s — all far into "Poor" (>4.0s); desktop LCP is comparatively better (1.2s–14.2s) but still fails "Good" on 2 of 3 desktop page types, and desktop isn't the traffic majority Google's field-data assessment weights most. Absolute ms values on mobile are somewhat inflated by this test environment's slower-than-calibrated host CPU (per Lighthouse's own warning), so treat exact numbers as directional — but the root-cause breakdown is unaffected: the LCP-breakdown insight attributes 16.5s of the 17.5s homepage LCP and 17.5s of the 18.5s product LCP to "resourceLoadDuration" (time spent downloading the oversized LCP image), not to TTFB (~380–420ms, fine) or render delay (<200ms). The LCP-discovery insight additionally flags that the LCP image is missing `fetchpriority="high"`. | Fix the root cause above (remove `unoptimized: true` + responsive sizing). Also verify hero/primary product images use `next/image`'s `priority` prop (which sets `fetchpriority="high"` and preloads automatically) — confirm it's applied to both the homepage hero and the product page's main gallery image. Once images are properly sized/compressed, LCP should drop close to or within the "Good"/"Needs Improvement" boundary without further changes.

**Homepage hero background is a 473KB unconverted PNG and is the page's LCP element** | High | The homepage LCP element is a full-bleed background `<img>` (`cdn-untitled-1-1-549dd1ae.png`). PNG's lossless compression is a poor fit for a photographic/gradient hero image — it's roughly 40% of the homepage's total 1.18MB desktop transfer weight on its own, and its download time dominates the homepage LCP timeline in both the mobile and desktop breakdowns. | Re-export the hero as WebP or AVIF (should shrink well below 100KB for equivalent visual quality) and confirm it's preloaded via `priority` so the browser doesn't have to discover it mid-render.

**INP proxy (Total Blocking Time) is "Needs Improvement" on mobile across all three page types** | Medium | Mobile TBT: homepage 430ms, category 300ms, product 420ms — all inside the 200–500ms "Needs Improvement" band for INP. Desktop TBT is good on all three (30–80ms). No third-party scripts were flagged as contributors, and per-page first-party JS is modest (~150–250KB across chunks), so this is most likely CPU contention from parsing/hydration compounding with concurrent image decode work during load, rather than one dominant long task. | Re-measure after the image-payload fix above — decoding fewer/smaller images during initial load typically recovers meaningful TBT for free. If it remains elevated, audit for client-side work during initial hydration (e.g., interactive product-grid/filter components) and consider deferring non-critical widgets with `next/dynamic`.

**Category page LCP element is text, yet render is delayed 3.7–9.8 seconds with no network dependency** | Medium | Unlike the homepage/product pages, the category page's LCP element is the `<h1>Shipping Containers</h1>` text node — no image fetch required. Yet elementRenderDelay is 9.8s on mobile and 3.7s on desktop even though TTFB is fine (~400–775ms), meaning something is blocking that text from painting well after the document is available. | Investigate what's blocking first paint on the category template specifically — check for render-blocking JS/CSS executing before hydration and confirm `font-display: swap` (or an equivalent) is applied to whatever custom font renders the `font-display` heading class, since a font-swap stall would produce exactly this symptom (fast TTFB, slow render of an otherwise-ready text node).
